# 分流解锁功能规划

## 1. 目标

本规划的目标是实现一套类似 soga 的“分流 + 解锁”能力，但不能只理解成简单的域名规则。完整能力应包含：

- 面板可配置：管理员可以维护分流策略、DNS 解析器、专用出站、解锁服务预设、节点能力标签。
- 后端可下发：后端能根据用户、套餐、节点、客户端能力生成稳定的策略配置。
- OmnXT 可执行：OmnXT 运行时能根据匹配结果选择直连、默认代理、拒绝、指定 DNS Resolver、指定 Outbound。
- 客户端可展示：OwlClient 能展示当前策略、服务解锁标签、测试结果，并将后端策略转换成 OmnXT 可识别的配置。
- 可观测和可回滚：每一次分流命中、出站选择、DNS 选择、失败回退都能被诊断，不应靠节点名称猜测是否支持解锁。

这里的“解锁”不是保证某个平台一定可用，而是提供一种可配置的流量调度能力：当访问 Netflix、Disney、OpenAI、TikTok、YouTube 等服务时，可以按规则走指定地区、指定节点、指定出站或者指定 DNS 策略，并提供健康检查、回退和诊断工具。

参考 soga 当前方向，soga 已经有路由出站和健康检查相关能力，例如 release 说明中提到“路由出站新增负载均衡健康检查，自动禁用无法连接的出站”。我们可以借鉴其目标形态，但实现上应贴合 NPanel + OwlClient + OmnXT 的现有结构。

参考链接：[soga releases](https://github.com/vaxilu/soga/releases)

## 2. 名词定义

### 2.1 分流规则

分流规则用于描述“什么流量应该走什么动作”。匹配条件包括：

- 域名后缀：如 `netflix.com`。
- 域名完整匹配：如 `api.openai.com`。
- 域名关键词：如 `youtube`。
- 域名正则：用于少量复杂场景。
- GeoSite：如 `geosite:netflix`、`geosite:openai`。
- GeoIP：如 `geoip:cn`。
- IP CIDR：如 `10.0.0.0/8`。
- 端口或端口范围：如 `443`、`8000-9000`。
- 国家或地区：如 `CN`、`US`。
- 进程、应用、入站来源：主要为桌面端或高级模式预留。

规则动作包括：

- `direct`：直连。
- `proxy`：走当前默认代理节点。
- `reject`：拒绝连接。
- `dns_resolver`：指定 DNS Resolver。
- `outbound`：指定出站标签。

### 2.2 解锁服务

解锁服务是对一组规则和目标出站的业务化封装。例如：

- Netflix：域名规则 + geosite 规则 + 指定 US/JP/HK 解锁出站。
- Disney：域名规则 + 指定出站。
- OpenAI：域名规则 + 指定 DNS + 指定出站。
- TikTok：域名规则 + 地区策略。

服务预设应是面板层概念，不应直接硬编码到客户端。客户端只消费后端下发的策略结果。

### 2.3 DNS Resolver

DNS Resolver 是可被规则引用的解析器。它需要描述：

- 协议：`udp`、`tcp`、`doh`、`dot`。
- 地址和端口。
- bootstrap DNS。
- 是否通过代理解析。
- 命中的域名范围。
- 超时、重试、健康状态。

DNS 策略要和出站策略分开配置，因为很多解锁失败不是代理不可用，而是 DNS 解析路径、污染、地区解析结果不一致导致。

### 2.4 Outbound

Outbound 是可被规则指定的出站目标。它可以对应：

- 某一个现有节点。
- 某一组候选节点。
- 一个专门的解锁节点。
- 一个外部中转配置。

Outbound 需要有稳定的 `tag`。规则只引用 `tag`，运行时通过 `tag` 找到真正的连接器。

### 2.5 Routing Profile

Routing Profile 是一组规则、DNS、Outbound、默认动作、回退策略的组合。不同套餐、用户组、节点、客户端版本可以绑定不同 Profile。

## 3. 当前现状

### 3.1 NPanel 后端现状

当前后端已经有节点配置入口：

- `api/admin/system/v1/system.proto` 中有 `GetNodeConfig` 和 `UpdateNodeConfig`。
- `NodeConfig` 已包含 `dns`、`block`、`outbound` 字段。
- `NodeDNS` 当前字段为 `proto`、`address`、`domains`。
- `NodeOutbound` 当前字段为 `name`、`protocol`、`address`、`port`、`password`、`rules`。
- biz 层的 `internal/biz/admin/system/system.go` 中，`NodeConfig.DNS`、`NodeConfig.Block`、`NodeConfig.Outbound` 仍是 JSON string，再由 `internal/service/admin/system/system.go` 转成 pb 结构。

节点服务端配置接口也已经有 DNS 和 Outbound 的基础结构：

- `api/server/v1/server.proto` 中有 `QueryServerProtocolConfig`。
- 服务端侧 `NodeDNS` 当前字段为 `server`、`domain`、`port`。
- 服务端侧 `NodeOutbound` 当前字段为 `tag`、`protocol`、`settings`。
- `internal/biz/server/node.go` 和 `internal/data/server_node.go` 已经负责节点侧配置查询、协议配置选择、订阅计划匹配、节点缓存清理。

订阅链路当前也很关键：

- `internal/biz/public/subscription/subscription.go` 通过 `subscribe_application` 的 `user_agent` 匹配客户端。
- `filterSubscriptionNodesByUserAgent` 当前只对官方客户端 UA 放行 `simnet`、`omniflow`、`mx` 等实验协议。
- `internal/biz/public/subscription/template.go` 通过模板函数渲染订阅，现有 OmnXT 相关模板函数主要围绕 SimNet 配置和链接。
- `ent/schema/proxysubscribeapplication.go` 的 `subscribe_template` 和 `output_format` 是订阅应用能力的核心存储位置。

这说明后端不是从零开始，但现在的结构还偏“节点运行配置”，不是完整的策略系统。它缺少：

- 分流 Profile。
- 规则优先级。
- 规则匹配类型。
- 规则动作。
- DNS Resolver 与规则之间的明确引用关系。
- Outbound 与节点、用户、套餐、服务解锁之间的绑定关系。
- 预览、测试、版本 hash、灰度、回滚。
- 独立 routing 领域的 ent schema、data repo、biz usecase、service、wire 注入、OpenAPI/Swagger 生成链。
- 针对官方客户端的增强配置协商机制。不能只依赖普通订阅模板，否则能力边界、缓存和客户端版本控制会变得很混乱。

### 3.2 OwlClient 现状

OwlClient 已有 OmnXT 配置生成骨架：

- `OmnxtConfigBuilder` 会生成 `dns`、`route`、`rule_sets`、`geoip_entries`、`geosite_entries`。
- 当前内置 direct 规则主要覆盖私网、回环和少量国家规则。
- `OmnxtRuleService` 当前主要从 `assets/omnxt_rules` 准备规则资源。
- 当前内置资源主要是 CN 相关 `geoip-cn`、`geosite-cn`。
- `OmnxtRule` 的 `RuleOutbound` 当前只有 `proxy`、`bypass`、`block`，不足以表达指定 DNS Resolver 和指定 Outbound。
- 插件客户端已有 `selectOutbound`、`testRule` 等接口，可作为诊断和调试基础。
- `Node` 模型当前已有 `country`、`city`、`tags`、`protocols`，但没有 routing profile、unlock service、outbound capability、route config hash 等字段。
- `SubscriptionRepository` 当前只获取订阅、节点、套餐，没有独立 routing config 获取接口。
- `OmnxtConnectionService` 当前启动时准备本地规则资源，再把 `ruleSets`、`geoipEntries`、`geositeEntries` 注入 `OmnxtConfigBuilder`。因此后端策略要接进启动链路，不能只新增 UI。
- 用户设置里已有 `proxyMode`，默认值为 `rule`，但它只是本地模式选择，不等于后端 Profile。

所以客户端已有规则注入的入口，但还没有完整消费后端 Profile 的数据模型和 UI。

### 3.3 NPanel 前端现状

NPanel 前端是 monorepo：

- 根目录用 `bun`、`turbo` 管理，admin 应用在 `apps/admin`。
- admin 前端使用 TanStack Router，路由文件位于 `apps/admin/src/routes/dashboard/...`，生成树为 `apps/admin/src/routeTree.gen.ts`。
- 导航配置位于 `apps/admin/src/layout/navs.ts`。
- 页面模块集中在 `apps/admin/src/sections/...`。
- API service 位于 `packages/ui/src/services/...`，配置从 `packages/ui/openapi2ts.config.ts` 读取 `docs/public/swagger/*.json` 生成。
- 当前很多 API 类型在 `packages/ui/src/services/gateway/typings.d.ts` 里仍是 `any`，新增复杂 routing 表单时不能完全依赖现有类型安全。

现有相关页面：

- `apps/admin/src/sections/system/index.tsx` 当前只挂基础设置、用户安全、日志设置；虽然 API service 有 `systemServiceGetNodeConfig` / `systemServiceUpdateNodeConfig`，但 UI 当前没有完整节点 DNS/Outbound 配置入口。
- `apps/admin/src/sections/subscribe/index.tsx` 只展示订阅配置和订阅应用协议配置。
- `apps/admin/src/sections/nodes/index.tsx` 和 `node-form.tsx` 当前支持节点名称、地址、端口、标签、节点组、节点类型、绑定服务器协议，但没有解锁服务、route role、健康检查等字段。
- `apps/admin/src/stores/node.ts` 和 `apps/admin/src/stores/server.ts` 已缓存节点、标签、服务器、协议端口，可被 routing/outbound 页面复用。

## 4. 目标架构

```text
NPanel Admin
  |
  | 维护分流 Profile、规则、DNS Resolver、Outbound、解锁服务预设
  v
NPanel Backend
  |
  | 生成版本化策略配置
  | 按用户、套餐、节点、客户端能力做裁剪
  v
Server API / Public API / Subscription
  |
  | OmnXT 客户端消费增强配置
  | 其他客户端继续消费兼容订阅
  v
OwlClient
  |
  | 转换为 OmnXT mobile config
  | 展示服务标签、策略状态、测试结果
  v
OmnXT Runtime
  |
  | rule match -> action -> direct/proxy/reject/dns/outbound
  v
实际网络连接
```

核心原则：

- 策略由后端管理，客户端不硬编码业务服务。
- OmnXT 是执行层，只执行明确配置。
- OwlClient 是展示和转换层，不负责猜测节点是否支持解锁。
- 兼容旧客户端，增强配置只下发给支持能力的客户端。
- 所有配置都有版本、hash、灰度和回滚路径。

### 4.1 `routing_profile` v1 冻结契约

本节作为 NPanel 后端、OmnXT、OwlClient、NPanel 前端共同遵守的第一版契约。后续开发优先按这里实现；如果需要破坏性调整，必须升级为 `routing_profile.v2`，不能在 v1 内静默改变字段含义。

#### 4.1.1 契约目标

`routing_profile.v1` 只解决第一版标准开发需要的稳定边界：

- 后端能按用户、套餐、用户组、节点、全局默认优先级生成最终策略。
- 配置能表达 DoH、DoT、UDP/TCP fallback、指定 DNS Resolver、指定 Outbound。
- Outbound 同时支持现有节点、节点组和 External Outbound。
- 服务解锁策略由后台控制，客户端只展示、诊断、测试和执行。
- 配置能表达失败回退、健康状态、能力要求和配置 hash。
- OmnXT SDK 能解析同一份 schema，OwlClient 能做 capability gate，节点端 control/state 能消费新版 `routing_profile`。
- 旧客户端、旧节点、旧订阅仍走原有 `dns/outbound/block` 或普通订阅，不被 v1 高级字段污染。

#### 4.1.2 顶层配置 Envelope

后端下发给 OwlClient / OmnXT SDK 的增强配置顶层对象统一使用：

```json
{
  "schema": "routing_profile.v1",
  "routing_version": 1,
  "routing_hash": "sha256:stable-canonical-config-hash",
  "generated_at": "2026-06-26T00:00:00Z",
  "expires_at": "2026-06-26T00:10:00Z",
  "mode": "enforce",
  "managed_by": "backend",
  "allow_user_override": false,
  "profile": {},
  "capability_requirements": {},
  "dns_resolvers": [],
  "outbounds": [],
  "unlock_services": [],
  "rules": [],
  "health_snapshot": {},
  "compat": {}
}
```

字段约定：

| 字段 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| `schema` | string | 是 | 固定为 `routing_profile.v1` |
| `routing_version` | integer | 是 | 单调递增版本，由后端生成 |
| `routing_hash` | string | 是 | 可执行配置的稳定 hash，格式 `sha256:<hex>` |
| `generated_at` | string | 是 | RFC3339 UTC 时间 |
| `expires_at` | string | 否 | 客户端缓存过期时间；过期后应尝试刷新，但不能立刻清空旧可用配置 |
| `mode` | string | 是 | `off`、`observe`、`enforce` |
| `managed_by` | string | 是 | 第一版固定为 `backend` |
| `allow_user_override` | boolean | 是 | 第一版默认 `false`，OwlClient 不提供绕过后台策略的本地开关 |
| `profile` | object | 是 | 当前最终命中的 Profile |
| `capability_requirements` | object | 是 | 执行该配置所需 OmnXT / OwlClient / 节点能力 |
| `dns_resolvers` | array | 是 | DNS Resolver 列表 |
| `outbounds` | array | 是 | Route Outbound 列表 |
| `unlock_services` | array | 是 | 服务预设和展示元数据 |
| `rules` | array | 是 | 已按优先级排序的规则 |
| `health_snapshot` | object | 否 | 后端聚合的最近健康状态快照，不参与 `routing_hash` |
| `compat` | object | 否 | 旧节点 / 旧客户端兼容信息 |

`routing_hash` 计算规则必须稳定：只包含影响执行结果的字段，例如 `mode`、`profile`、`capability_requirements`、`dns_resolvers`、`outbounds`、`unlock_services`、`rules`、`compat`。不包含 `generated_at`、`expires_at`、`health_snapshot`、后端内部自增 ID、展示-only 文案更新时间。JSON canonicalization 建议使用稳定 key 排序、空数组不省略、空对象不省略。

#### 4.1.3 Profile 对象

`profile` 对象定义当前用户最终命中的策略：

```json
{
  "id": "profile_123",
  "code": "default_smart",
  "name": "默认智能分流",
  "description": "默认用户级分流策略",
  "scope": {
    "type": "user",
    "id": "10001"
  },
  "priority": 10,
  "default_action": {
    "type": "proxy"
  },
  "default_dns_resolver_tag": "dns:system",
  "default_fallback_policy": "fallback_default",
  "enabled": true
}
```

`scope.type` 第一版支持：

- `user`
- `plan`
- `group`
- `node`
- `global`

Profile 命中优先级固定为：

```text
用户专属 Profile
  > 套餐 Profile
  > 用户组 Profile
  > 节点 Profile
  > 全局默认 Profile
```

如果同一层级多个 Profile 同时命中，按 `priority` 升序选择；`priority` 相同则按 `updated_at` 最新优先。后端 preview 和实际下发必须使用同一套算法，不能出现预览命中和实际下发不一致。

#### 4.1.4 Capability Requirements

`capability_requirements` 用于让 OwlClient 和 OmnXT SDK 判断“能否执行”，不能只靠 UA 字符串猜测：

```json
{
  "min_schema": "routing_profile.v1",
  "min_omnxt_sdk": "0.1.0",
  "min_flutter_plugin": "0.1.0",
  "required_features": [
    "route_outbound",
    "route_dns_resolver",
    "doh",
    "dot"
  ],
  "optional_features": [
    "route_events",
    "external_wireguard",
    "external_socks",
    "external_http",
    "client_health_report",
    "node_health_report_v1"
  ],
  "node_required_features": [
    "node_routing_profile_v1",
    "node_health_report_v1"
  ]
}
```

能力标识第一版固定如下：

| 能力 | 说明 | 不满足时行为 |
| --- | --- | --- |
| `route_outbound` | 按规则选择指定 outbound connector | 不执行 routing，只展示不可用诊断 |
| `route_dns_resolver` | 按规则选择指定 DNS resolver | 不执行 DNS 分流，只展示不可用诊断 |
| `route_events` | 上报命中、回退、健康变化事件 | 可执行，但诊断页标记事件能力缺失 |
| `doh` | 支持 DoH resolver | 含 DoH 配置时不可执行 |
| `dot` | 支持 DoT resolver | 含 DoT 配置时不可执行 |
| `external_wireguard` | 支持 WireGuard External Outbound | 含该 outbound 时不可执行对应规则 |
| `external_socks` | 支持 SOCKS External Outbound | 含该 outbound 时不可执行对应规则 |
| `external_http` | 支持 HTTP External Outbound | 含该 outbound 时不可执行对应规则 |
| `client_health_report` | 客户端本机 RTT / URL / DNS 可达性测试 | 可执行，但 health 来源缺少本机视角 |
| `node_routing_profile_v1` | 节点端能消费新版 routing profile | 旧节点只能走兼容结构 |
| `node_health_report_v1` | 节点端能上报 resolver/outbound/service health | 后端不能把该节点声明为支持节点端健康 |

`required_features` 必须由后端根据最终配置实际生成：配置里出现 DoH 才要求 `doh`，出现 DoT 才要求 `dot`，出现 External WireGuard 才要求 `external_wireguard`。`route_events` 默认放入 `optional_features`，因为缺少事件能力不应阻止执行分流；如果某个企业策略要求“无诊断事件不执行”，可以在后续 v1 扩展中增加策略字段显式提升为 required，不能让客户端自行猜测。

客户端请求 routing config 时，建议携带：

```json
{
  "client": "owlclient",
  "client_version": "x.y.z",
  "platform": "android",
  "omnxt_sdk_version": "x.y.z",
  "flutter_plugin_version": "x.y.z",
  "supported_schemas": ["routing_profile.v1"],
  "supported_features": [
    "route_outbound",
    "route_dns_resolver",
    "route_events",
    "doh",
    "dot"
  ]
}
```

后端必须根据 `supported_features` 裁剪配置。不能把客户端明确不支持的 External Outbound、DNS 协议或 action 静默下发后再期待客户端忽略。

#### 4.1.5 保留 Tag 和命名空间

v1 需要固定内置 tag，避免示例和实现出现“引用了不存在对象”的歧义：

| tag | 类型 | 是否必须出现在数组中 | 含义 |
| --- | --- | --- | --- |
| `proxy:default` | outbound | 否 | 用户当前默认代理，即当前 `selected_outbound` |
| `dns:system` | dns resolver | 否 | SDK / 系统默认 DNS 策略 |

保留 tag 可以被 `default_dns_resolver_tag`、`fallback_pool_tags`、rule action 引用，即使它们没有出现在 `dns_resolvers` 或 `outbounds` 数组中。实现时可以把它们物化为内部对象，但对外语义必须保持稳定。

自定义 tag 命名空间建议：

- `dns:<name>`：DNS Resolver。
- `unlock:<service>:<region>`：服务解锁 Outbound。
- `external:<protocol>:<name>`：External Outbound。
- `node:<node_id>`：显式节点 Outbound。
- `group:<node_group_id>`：显式节点组 Outbound。

后台校验必须禁止管理员创建与保留 tag 冲突的自定义 tag。

#### 4.1.6 DNS Resolver Schema

DNS Resolver 对象统一使用：

```json
{
  "tag": "dns:cloudflare-doh",
  "name": "Cloudflare DoH",
  "proto": "doh",
  "address": "https://cloudflare-dns.com/dns-query",
  "port": 443,
  "path": "/dns-query",
  "server_name": "cloudflare-dns.com",
  "bootstrap": ["1.1.1.1", "1.0.0.1"],
  "detour": {
    "type": "proxy"
  },
  "timeout_ms": 3000,
  "retry": 2,
  "cache_ttl_seconds": 300,
  "health_check": {
    "enabled": true,
    "domain": "www.cloudflare.com",
    "interval_seconds": 60
  },
  "enabled": true
}
```

字段约定：

| 字段 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| `tag` | string | 是 | 稳定引用标签，必须全局唯一 |
| `proto` | string | 是 | `doh`、`dot`、`udp`、`tcp` |
| `address` | string | 是 | DoH 使用完整 HTTPS URL；DoT/UDP/TCP 使用 host 或 IP |
| `port` | integer | 否 | DoH 默认 443，DoT 默认 853，UDP/TCP 默认 53 |
| `path` | string | 否 | DoH path |
| `server_name` | string | 否 | DoH/DoT SNI 或证书校验名称 |
| `bootstrap` | array | 否 | 解析 resolver hostname 时使用的 bootstrap DNS |
| `detour` | object | 是 | DNS 请求自身走 `direct`、`proxy` 或指定 outbound |
| `timeout_ms` | integer | 是 | 单次查询超时 |
| `retry` | integer | 否 | 重试次数 |
| `cache_ttl_seconds` | integer | 否 | 客户端缓存 TTL 上限 |
| `health_check` | object | 否 | 健康检查配置 |
| `enabled` | boolean | 是 | 是否启用 |

`detour` Schema：

```json
{ "type": "direct" }
{ "type": "proxy" }
{ "type": "outbound", "tag": "unlock:openai:us" }
```

第一版正式支持 `DoH` 和 `DoT`。`UDP` / `TCP` 只作为兼容和 fallback，管理端可以提供，但不能把第一版能力降级成只支持 UDP/TCP。

#### 4.1.7 Route Outbound Schema

Route Outbound 对象统一使用：

```json
{
  "tag": "unlock:openai:us",
  "name": "OpenAI US",
  "type": "node",
  "node_id": "node_100",
  "node_group_id": null,
  "region": "US",
  "service_tags": ["openai"],
  "selection_policy": "single",
  "fail_policy": "fallback_default",
  "fallback_pool_tags": ["unlock:openai:sg", "proxy:default"],
  "health_check": {
    "enabled": true,
    "url": "https://chat.openai.com/cdn-cgi/trace",
    "interval_seconds": 60,
    "timeout_ms": 5000
  },
  "enabled": true
}
```

`type` 第一版支持：

- `node`：复用现有单个节点。
- `node_group`：复用现有节点组，由 `selection_policy` 选择候选节点。
- `external`：使用独立外部出站。

`selection_policy` 第一版支持：

- `single`
- `round_robin`
- `lowest_latency`
- `health_first`
- `manual`

External Outbound 使用同一个对象，增加 `external` 子对象：

```json
{
  "tag": "external:wg:jp-01",
  "name": "WG JP 01",
  "type": "external",
  "region": "JP",
  "service_tags": ["netflix", "disney"],
  "external": {
    "protocol": "wireguard",
    "sensitive": true,
    "delivery": "enhanced_config_only",
    "settings": {
      "endpoint_host": "203.0.113.10",
      "endpoint_port": 51820,
      "public_key": "base64-public-key",
      "private_key": "base64-private-key",
      "address": ["10.0.0.2/32"],
      "allowed_ips": ["0.0.0.0/0", "::/0"],
      "dns": ["1.1.1.1"],
      "mtu": 1280,
      "persistent_keepalive_seconds": 25
    }
  },
  "fail_policy": "fallback_default",
  "enabled": true
}
```

External Outbound 第一版协议固定为：

| 协议 | `external.protocol` | 必要字段 |
| --- | --- | --- |
| WireGuard | `wireguard` | `endpoint_host`、`endpoint_port`、`public_key`、`private_key`、`address`、`allowed_ips` |
| SOCKS | `socks` | `host`、`port`、`version`，可选 `username`、`password` |
| HTTP | `http` | `host`、`port`，可选 `username`、`password`、`tls`、`headers` |

`external.delivery` 第一版支持：

| delivery | 含义 |
| --- | --- |
| `enhanced_config_only` | 只下发给通过 capability gate 的官方客户端增强配置 |
| `node_control_only` | 只下发给 OmnXT 节点控制面，OwlClient 只能看到脱敏状态或完全裁剪 |

安全约束：

- External Outbound 凭据只能出现在增强 routing config 或节点控制面，不允许进入普通订阅模板。
- 后端日志、审计日志、preview 响应默认必须脱敏 `private_key`、`password`、`headers.authorization` 等敏感字段。
- 如果 External Outbound 只允许节点端使用，客户端下发时应使用 `delivery=node_control_only` 或直接裁剪该 outbound。
- OwlClient 缓存包含敏感 External Outbound 时必须使用安全存储或加密数据库，不能落入普通明文日志。

#### 4.1.8 Unlock Service Schema

服务预设对象只描述业务展示和默认策略，不把服务域名硬编码在客户端：

```json
{
  "code": "openai",
  "name": "OpenAI",
  "category": "ai",
  "regions": ["US", "SG", "JP"],
  "default_region": "US",
  "default_outbound_tag": "unlock:openai:us",
  "default_dns_resolver_tag": "dns:cloudflare-doh",
  "default_fail_policy": "fallback_default",
  "health_check_url": "https://chat.openai.com/cdn-cgi/trace",
  "enabled": true
}
```

服务预设必须来自数据库或配置，可以初始化内置默认值，但新增、修改、禁用服务都应走后台配置，不允许在 OwlClient 或 OmnXT 里写死服务列表。

#### 4.1.9 Rule Schema

规则对象统一使用：

```json
{
  "id": "rule_openai_us",
  "name": "OpenAI US",
  "priority": 200,
  "enabled": true,
  "service_code": "openai",
  "matcher": {
    "type": "domain_suffix",
    "value": "openai.com"
  },
  "action": {
    "type": "outbound",
    "outbound_tag": "unlock:openai:us",
    "dns_resolver_tag": "dns:cloudflare-doh",
    "fail_policy": "fallback_default"
  }
}
```

`matcher.type` 第一版支持：

- `domain_suffix`
- `domain_full`
- `domain_keyword`
- `domain_regex`
- `geosite`
- `geoip`
- `ip_cidr`
- `country`
- `port`
- `port_range`
- `process`
- `app`
- `inbound_source`
- `compound`

简单 matcher 使用：

```json
{ "type": "domain_suffix", "value": "netflix.com" }
```

复合 matcher 使用：

```json
{
  "type": "compound",
  "op": "and",
  "items": [
    { "type": "domain_suffix", "value": "openai.com" },
    { "type": "port", "value": 443 }
  ]
}
```

`compound.op` 支持 `and`、`or`、`not`。`not` 只能包含一个 item。

`action.type` 第一版支持：

- `direct`
- `proxy`
- `reject`
- `dns_resolver`
- `outbound`

Action 字段约定：

| action | 必填字段 | 说明 |
| --- | --- | --- |
| `direct` | 无 | 直连 |
| `proxy` | 无 | 使用用户当前默认代理，即当前 `selected_outbound` |
| `reject` | 可选 `reason` | 拒绝连接 |
| `dns_resolver` | `dns_resolver_tag` | 仅指定 DNS 查询路径 |
| `outbound` | `outbound_tag` | 指定连接出站；可同时带 `dns_resolver_tag` 指定 DNS |

规则优先级越小越先匹配。后端下发前必须完成排序，OwlClient 和 OmnXT 不应重新解释排序规则。

#### 4.1.10 失败策略契约

`fail_policy` 第一版支持：

| 策略 | 含义 | 适用场景 |
| --- | --- | --- |
| `fallback_default` | 优先回到同服务候选池，再回到用户当前默认代理 | 解锁服务默认策略 |
| `fallback_pool` | 只在 `fallback_pool_tags` 内回退 | 多地区服务候选池 |
| `fallback_direct` | 回到直连 | CN direct、私网、企业内网 |
| `fail_closed` | 拒绝连接并上报诊断事件 | 安全、隐私、强风控服务 |
| `observe_only` | 只记录命中和失败，不改变真实流量 | 灰度和预览 |
| `disable` | 后端或健康检查禁用该 outbound/rule | 自动降权或人工禁用 |

解锁和地区敏感服务默认必须使用 `fallback_default`，不能默认 `fallback_direct`，避免真实 IP 泄漏或地区错误。

回退事件必须至少包含：

```json
{
  "event": "routing_fallback",
  "profile_code": "default_smart",
  "routing_hash": "sha256:stable-canonical-config-hash",
  "rule_id": "rule_openai_us",
  "service_code": "openai",
  "from_action": {
    "type": "outbound",
    "outbound_tag": "unlock:openai:us"
  },
  "to_action": {
    "type": "proxy"
  },
  "reason": "outbound_unhealthy",
  "occurred_at": "2026-06-26T00:00:00Z"
}
```

#### 4.1.12 Public API 和 Server API 契约

官方客户端推荐新增独立 public 接口：

```text
GET /v1/public/routing/config
POST /v1/public/routing/preview
POST /v1/public/routing/test
```

`GET /v1/public/routing/config` 输入来源：

- 登录态 Bearer token，或后续明确的订阅 token。
- `User-Agent`。
- 客户端 capability 请求头或 JSON 参数。
- 当前用户、套餐、节点列表、订阅状态由后端解析，不由客户端自行拼装。

建议请求头：

```text
X-Routing-Schema: routing_profile.v1
X-OmnXT-SDK-Version: x.y.z
X-OwlClient-Version: x.y.z
X-Routing-Features: route_outbound,route_dns_resolver,route_events,doh,dot
If-None-Match: sha256:stable-canonical-config-hash
```

返回：

- `200`：返回完整 `routing_profile.v1`。
- `304`：hash 未变化，客户端继续使用本地缓存。
- `409`：客户端能力不足，返回缺失 capability 列表。
- `422`：用户无可用 Profile 或后端配置校验失败。

OmnXT 节点端推荐在 `/v2/server` 或新版节点端 endpoint 中携带：

- `routing_profile`：新版 schema。
- `routing_hash`：配置 hash。
- `routing_capabilities`：节点应具备或已启用能力。
- `legacy_dns` / `legacy_outbound`：旧节点兼容字段，保留旧行为。

P0 可以只确认字段和 hash，不要求节点端完整执行；标准开发阶段必须让 `crates/omnxt-control`、`crates/omnxt-state` 消费该 schema。

## 5. 数据模型规划

### 5.1 Routing Profile 表

建议新增 `routing_profiles`：

| 字段 | 说明 |
| --- | --- |
| `id` | 主键 |
| `name` | 展示名称，如“默认智能分流” |
| `code` | 稳定编码，如 `default_smart` |
| `description` | 说明 |
| `mode` | `off`、`observe`、`enforce` |
| `default_action` | `proxy`、`direct`、`reject` |
| `dns_mode` | `system`、`direct`、`proxy`、`rule` |
| `fallback_action` | 出站失败后的动作 |
| `priority` | Profile 选择优先级 |
| `scope_type` | `global`、`group`、`plan`、`user`、`node` |
| `scope_id` | 作用对象 ID |

| `version` | 配置版本 |

`observe` 模式用于灰度：后端和客户端可以记录“如果启用会命中哪条规则”，但不改变实际出站。

### 5.2 Routing Rule 表

建议新增 `routing_rules`：

| 字段 | 说明 |
| --- | --- |
| `id` | 主键 |
| `profile_id` | 所属 Profile |

| `matcher_type` | 匹配类型 |
| `matcher_value` | 匹配值 |
| `matcher_json` | 复杂 `and/or/not` 表达式 |

| `remark` | 备注 |
| `enabled` | 是否启用 |

`matcher_type` 建议支持：

- `domain_suffix`
- `domain_full`
- `domain_keyword`
- `domain_regex`
- `geosite`
- `geoip`
- `ip_cidr`
- `country`
- `port`
- `port_range`
- `process`
- `app`
- `inbound_source`
- `compound`

### 5.3 DNS Resolver 表

建议新增 `routing_dns_resolvers`：

| 字段 | 说明 |
| --- | --- |
| `id` | 主键 |
| `tag` | 稳定标签，如 `dns:cloudflare-doh` |
| `name` | 展示名称 |
| `proto` | `udp`、`tcp`、`doh`、`dot` |
| `address` | 地址 |
| `port` | 端口 |
| `path` | DoH path |
| `bootstrap` | bootstrap DNS |
| `detour` | `direct`、`proxy`、`outbound:<tag>` |

### 5.4 Outbound 表

建议新增 `routing_outbounds`：

| 字段 | 说明 |
| --- | --- |
| `id` | 主键 |
| `tag` | 稳定标签，如 `unlock:netflix:us` |
| `name` | 展示名称 |
| `type` | `node`、`node_group`、`external` |
| `server_id` | 绑定服务器 |
| `node_id` | 绑定节点 |
| `protocol` | 协议，节点复用时取现有节点协议，External Outbound 支持 `wireguard`、`socks`、`http` |
| `settings_json` | 协议设置 |
| `region` | 地区，如 `US`、`JP`、`HK` |
| `service_tags` | 支持服务标签 |
| `health_check_url` | 健康检查 URL |
| `fail_policy` | `disable`、`fallback_default`、`fail_closed` |
| `enabled` | 是否启用 |

Outbound 已确认需要同时支持两种形态：

- 默认复用现有节点或节点组作为 route outbound，便于管理员直接把已有节点标记为 OpenAI、Netflix、TikTok 等服务可用出站。
- 允许配置 External Outbound，第一批建议支持 `WireGuard`、`SOCKS`、`HTTP`。External Outbound 的凭据、私钥、header、认证信息必须只进入增强 routing config 或节点控制面，不允许暴露到普通订阅模板。

如果前期不想引入复杂表，可以先把 Outbound 和 Resolver 放在系统配置 JSON 中，但从长期看，独立表更利于权限、列表、测试、灰度和审计。

| `default_dns_resolver_id` | 默认 DNS |
| `enabled` | 是否启用 |

服务预设只提供默认规则，不应限制管理员新增自定义域名和自定义动作。

### 5.6 节点能力扩展

建议给节点或协议配置增加能力字段：

| 字段 | 说明 |
| --- | --- |

| `country` | 国家码 |
| `city` | 城市 |
| `capability_tags` | `unlock`、`streaming`、`ai`、`dns` |

- 如果为了早期 PoC 复用订阅模板，则只能新增模板函数输出 OmnXT routing JSON，并且必须限制在官方 UA 和指定 `subscribe_application` 行里。

结合当前 OwlClient 实际调用链，必须特别注意：

- OwlClient 当前取用户订阅走 `GET /v1/public/user/subscribe`。
- OwlClient 当前取节点列表走 `GET /v1/public/subscribe/node/list`，对应 `api/public/subscribe/v1/subscribe.proto` 的 `QueryUserSubscribeNodeList`。
- 新增 routing config 不能只挂在传统 `/v1/subscribe/config` 或 `api/public/subscription/v1` 订阅生成接口，否则登录态客户端拿节点列表时不会自动拿到策略。
- 建议新增 `api/public/routing/v1` 独立接口，并在 OwlClient `ApiClient` 抽象增加方法；如果短期把 routing summary 附在节点列表中，也必须同步 `UserSubscribeNodeInfo`、`internal/biz/public/subscribe`、`internal/service/public/subscribe`、`internal/data/public_subscribe.go`、前端/客户端模型。
- `internal/server/http.go` 当前已注册 public subscribe 和 public subscription HTTP 服务；`internal/server/grpc.go` 注册范围更窄。如果 routing 只给 HTTP 客户端使用，可先不暴露 gRPC，但文档和实现要明确，避免只改 HTTP 后误以为 gRPC 也可用。
- `ServiceContext.Auth()` 当前匿名放行的是 `/v1/auth/`、`/v1/common/`、`/v1/public/portal/`、`/v1/server/`、`/v2/server/` 等前缀，并没有放行 `/v1/public/routing/`。如果 routing config 设计为登录态 Bearer 接口，可以保留认证；如果设计为订阅 token / 设备预登录接口，必须显式补 `NoAuthPaths` 或匿名前缀，并加鉴权测试。
- `DeviceMiddleware` 当前只处理 `/v1/public/subscribe/`、`/v1/public/user/`、`/v1/public/order/` 等前缀，没有处理 `/v1/public/routing/`。如果官方客户端在 device login / 加密模式下调用 routing 接口，需要决定是否把 routing 纳入设备加密链路；否则客户端会出现“节点接口可用、routing 接口解密/验签不一致”的隐性问题。
- `LegacyRouteGuardFilter` 当前没有拦截 `/v1/public/routing/`，`SubscribePathCompatFilter` 主要保护自定义订阅路径下的 `/v1/subscribe/*`。新增 routing 路由时仍要补路由级回归测试，避免后续 custom subscribe path 或 legacy guard 调整时误伤官方客户端。
- OwlClient 全局 Dio 已通过 `DefaultUserAgentInterceptor` 自动补 `User-Agent: Slag/<version> (...) AnyiNet/2.0`，Xboard/Xiaov2board 还会用面板返回的 `ua` 覆盖默认 UA。后端按 UA 做 capability 判断时，必须兼容这两类 UA 来源，不能只匹配 Flutter app 内部 `DeviceInfoService.userAgent` 的格式。
- `configs/config.yaml` 和 `configs/config.docker.yaml` 都维护 `server.auth.no_auth_paths`、`server.cors`、`device` 等关键配置。routing 接口如果涉及匿名、订阅 token、设备加密或跨域访问，两个配置样例都要同步更新，并补 `internal/middleware/auth_test.go`、`internal/pkg/middleware/device_test.go`、`internal/pkg/middleware/cors_test.go` 的覆盖。
- `internal/pkg/middleware/pandomain.go` 是泛域名订阅中间件，核心职责是通过自定义域名和 token 返回订阅配置。routing 增强接口不要默认挂进这条链路，除非明确设计成 token 订阅域名的一部分，否则容易让自定义订阅域名返回 routing JSON 或让普通订阅路径被污染。
- `internal/server/request_decoder.go` 和 `internal/server/encoder.go` 说明当前请求解码、响应编码存在路径级兼容逻辑，`internal/server/encoder_test.go` 还专门测试了 `EmitUnpopulated`。routing config 如果要求空数组、空对象稳定下发，或者 preview/test 接口接受复杂 JSON，必须补请求解码和响应 shape 单测，不能只测业务 usecase。

### 6.3 数据校验

必须做强校验：

- `action=outbound` 时，必须引用存在且启用的 Outbound。
- `action=dns_resolver` 时，必须引用存在且启用的 Resolver。
- 同一个 Profile 下规则优先级不能冲突，或者必须有明确排序。
- `domain_regex` 必须限制长度和复杂度，避免高 CPU。
- `settings_json` 必须按协议校验，不允许任意下发。
- 禁止把后台敏感字段直接暴露到通用订阅。
- 不支持 OmnXT routing 的客户端不应收到高级结构。

### 6.4 健康检查和回退

建议后端增加定时任务：

- 对 Outbound 做 TCP 连接测试。
- 对流媒体服务做轻量 URL 测试。
- 对 DNS Resolver 做指定域名解析测试。
- 记录最近失败原因、延迟、连续失败次数。
- 连续失败后根据 `fail_policy` 自动禁用或降级。

回退策略建议：

- `fallback_default`：回到当前默认代理节点。
- `fallback_direct`：回到直连。
- `fail_closed`：拒绝连接，适合风控严格服务。
- `observe_only`：只记录，不影响用户流量。

推荐默认策略采用偏“大厂风格”的保守降级：

- 对 OpenAI、Netflix、Disney、TikTok 等解锁或地区敏感服务，默认 `fail_policy=fallback_default`，即主 outbound 不可用时回到当前用户默认代理或同服务候选池，不默认直连，避免真实 IP 泄漏和地区错误。
- 对安全、隐私、企业策略或强风控服务，允许管理员显式配置 `fail_closed`。此时出站不可用就拒绝连接，并给客户端诊断事件，不做静默降级。
- `fallback_direct` 只建议用于 CN direct、私网、局域网、企业内网等本来就应直连的规则，不作为解锁服务默认回退。
- 支持多级回退：`primary outbound` -> `同服务候选 outbound pool` -> `用户当前默认代理` -> `fail_closed/observe_only`。每次回退必须记录原因、耗时、命中的 rule/profile/hash。

### 6.5 订阅模板兼容

NPanel 已有订阅应用和模板机制，分流解锁必须和订阅渲染打通：

- `OwlClient/OmnXT` 使用增强配置。
- 传统客户端使用原模板，不要下发无法识别的字段。
- 管理员可以在订阅应用里选择是否启用分流增强。
- UA 识别必须明确，不能只靠模糊字符串。
- 需要预览某个用户最终拿到的订阅和 routing config。

结合当前代码，订阅兼容要特别处理：

- `matchSubscribeApplication` 以 `user_agent` 子串匹配，且第一个命中的应用会被选中；新增 OwlClient/OmnXT 应用行时要确保 UA 比较具体，避免被默认行或泛化行提前命中。
- `ValidateLegacyRequest` 会把 `subscribe_application.user_agent` 加入允许 UA 范围；如果开启 `UserAgentLimit`，官方客户端 UA 必须同步配置。
- `filterSubscriptionNodesByUserAgent` 当前只对官方客户端放行实验协议。routing 增强也应复用这个“官方客户端能力门”，但不要把“官方客户端”直接等同于“支持 routing”，最好新增明确 capability。
- `RenderTemplate` 现在只有 `buildOmnxtSimnetConfigs`、`buildOmnxtProtocolLinks` 这类模板函数；如果走模板 PoC，需要新增类似 `buildOmnxtRoutingConfig` 的函数，并配测试。
- `PreviewSubscribeTemplate` 当前只预览订阅模板本身，不知道具体用户、Profile、客户端能力。后续需要新增 routing preview，不要把 routing 预览挤进普通模板预览。

### 6.6 数据迁移

现有 `NodeConfig.dns`、`NodeConfig.outbound` 可以迁移为默认 Profile：

- `NodeDNS` 迁移到 `routing_dns_resolvers`。
- `NodeOutbound` 迁移到 `routing_outbounds`。
- `NodeOutbound.rules` 迁移到 `routing_rules`。
- 迁移后保留旧字段一段时间，作为兼容输出。

推荐迁移策略：

1. 新表上线，但旧配置仍可读写。
2. 管理端展示新 UI，同时支持从旧配置导入。
3. 节点和客户端使用 `routing_hash` 判断是否支持新配置。
4. 稳定后再逐步弱化旧字段。

迁移和种子还要注意当前仓库的两条初始化路径：

- 新库初始化主要走 `AutoMigrateWithData` / `AutoMigrate`，ent schema 会自动建表，但不会自动写入默认分流 Profile、默认 DNS、默认 Outbound、内置解锁服务等业务种子。
- 旧库补丁 SQL 走 `internal/migrate/legacy_seed.go` 内的 `legacySQLMigrations` 白名单。若新增默认预设或旧字段导入 SQL，除了添加 `internal/migrate/sql/*.sql`，还必须把文件名登记到 `legacySQLMigrations`，否则线上旧库不会执行。
- 默认预设应具备幂等写入能力，推荐用稳定 `code/tag` 做唯一键，例如 `default_smart`、`dns:system`、`proxy:default`，避免重复初始化。
- ent schema 里要显式声明唯一约束和排序索引，例如 profile code、outbound tag、resolver tag、profile rule priority、service code；不要只依赖应用层检查。

### 6.7 代码生成和缓存清理

当前仓库的 proto、ent、wire、swagger 都是生成链路的一部分，routing 功能不能只改 Go 源码。

需要补齐：

- ent schema 变更后运行 ent 代码生成，并检查迁移 SQL。
- proto 变更后运行 `make api`，重新生成 pb/http/grpc/validate/openapi 代码。
- wire provider 变更后运行 `make generate`，重新生成 `cmd/npanel/wire_gen.go` 并执行 `go mod tidy`。
- swagger/openapi 变更后刷新 `docs/public/swagger/*.json`，再让前端生成 `packages/ui/src/services/...`。
- Profile、Rule、Resolver、Outbound 修改后，要清理相关节点和订阅缓存；当前 node 更新已有 `ClearNodeCache` 模式，routing 模块也需要同等级缓存失效。
- routing config hash 必须由规范化后的最终配置生成，不能直接 hash 数据库行，否则规则排序或 JSON 字段顺序可能导致无意义变更。
- Redis / cache key 建议独立命名空间，例如 `routing:profile:{id}`、`routing:compiled:{user_subscribe_id}:{client_cap_hash}`、`routing:hash:{scope}`，不要复用现有 `node:*`、`system:*` key，避免清缓存时互相污染。
- 缓存失效触发点至少包含：Profile 变更、Rule 排序变更、DNS Resolver 变更、Outbound 变更、Unlock Service capability 变更、节点状态/能力变更、套餐或用户组绑定变更、`subscribe_application` capability 变更。
- 当前仓库已有 `api/admin/authmethod/v1/compat_validate.go` 这类兼容 shim，说明历史上 `make api` 与 `*.pb.validate.go` 曾出现过不同步问题。routing proto 如果引入 validate 规则，必须确认生成后的 `*.pb.validate.go` 没有引用旧字段；不能只看到 `make api` 成功就跳过编译和测试。
- `legacy_seed_test.go` 会检查已同步的 legacy SQL 版本列表。新增 routing 相关 legacy SQL 后，要同步补测试期望或新增单测，避免文件存在但没有进入 `legacySQLMigrations` 白名单。

落地时建议把以下命令列入 PR 检查：

```bash
make api
make generate
go test ./internal/biz/... ./internal/data/... ./internal/service/... ./internal/server/...
go test ./internal/migrate ./internal/pkg/middleware ./api/...
```

如果新增 `api/public/routing/v1`，还要确认 `openapi.yaml` 和 `docs/public/swagger/user.json` 或拆分后的 swagger 文件包含该接口，否则前端 `bun run openapi` 生成不到 service。

## 7. OmnXT 需要修改的部分

### 7.0 核实结论：SDK 端必改，节点端标准开发必改

复查 OmnXT 代码后，结论如下：

- 如果目标是 OwlClient 上实现“访问不同域名自动走不同节点 / 出站 / DNS”，OmnXT 必须修改，而且主修改点在 SDK 端和客户端运行时。
- 只改 OwlClient Dart 层不够。Dart 可以生成规则配置，但当前 native SDK 运行时仍主要围绕单个 `selected_outbound` 构造 connector，无法保证每条连接按 `RuleAction::Outbound(tag)` 选择不同 connector。
- 从纯客户端 PoC 看，节点端不是验证“域名命中后本地 SDK 选择不同 connector”的硬依赖。客户端命中规则后选择哪个出站，本质发生在用户设备上的本地代理 / TUN / SDK 运行时，远端节点只看到普通入站连接。
- 但产品范围已经确认第一版标准开发要支持后端新版 `routing_profile`，并且节点端需要做解锁服务探测和健康上报。因此节点端不再是“可永久延后”的能力，只能在最小 PoC 中临时不依赖，正式 P1-P6 开发必须包含 control/ppanel 适配和节点健康上报。
- 节点端已有一部分 PPanel 控制面基础：`PPanelServerConfigData` 已有 `dns`、`outbound` 字段，`PPanelSnapshotAssembler` 会把它们转成 `Dns` / `Routes` bundle，`omnxt-state` 也能把这些 bundle 转为 rule_sets 和 control_plane_outbounds。
- 但是，现有节点端支持的是旧 PPanel 风格的 `dns/outbound/block`，不是本文规划的 `routing_profiles/routing_rules/unlock_services/capabilities/health` 新模型。如果后端要让 OmnXT 节点直接消费新版 routing schema，节点端 control/ppanel 适配也要改。
- `crates/omnxt-sdk-uniffi` 当前已经暴露 `omnxt_mobile_capabilities`、`omnxt_mobile_parse_config_json`、`omnxt_mobile_build_config_json`、`OmnxtMobileService.apply_rule_set_json`、`test_rule_match_json`、`select_outbound`、`auto_select_outbound` 等入口。
- Flutter 插件 `apps/FlutterOmnxtPlugin/lib/src/omnxt_plugin.dart` 已封装 `applyRuleSet`、`testRule`、`selectOutbound`。这些入口说明“诊断/手动切换”已有基础，但不能替代每条连接按规则自动选择 connector 的运行时闭环。
- 还必须把 `crates/omnxt-sdk-cdylib` 纳入修改范围。OwlClient 实际依赖 Flutter 插件里的动态库和 C ABI，`omnxt-sdk` / `omnxt-sdk-uniffi` 改完后，如果 `omnxt-sdk-cdylib/src/lib.rs`、生成头文件和 Flutter FFI wrapper 没刷新，客户端仍会调用旧 ABI。
- `crates/omnxt-sdk-cdylib` 当前已经导出 `omnxt_mobile_capabilities_json`、`omnxt_mobile_parse_config_json`、`omnxt_mobile_build_config_json`、`omnxt_mobile_service_apply_rule_set_json`、`omnxt_mobile_service_select_outbound_json`、`omnxt_mobile_service_auto_select_outbound_json`、`omnxt_mobile_service_groups_json` 等函数。新增 routing schema 或 runtime API 时，必须同步 C ABI、Dart FFI、Android/iOS/macOS/Windows/Linux artifact。

因此建议分两阶段：

1. 第一阶段做客户端闭环：改 OmnXT SDK / Flutter plugin / OwlClient，使客户端可以按规则切换出站和 DNS。
2. 标准开发阶段做节点控制面增强：NPanel 后端下发新版 routing schema 后，OmnXT control/ppanel/node 侧必须识别 `routing_profile`、执行节点端健康探测，并把 resolver/outbound/service health 回传后端。

### 7.1 配置 Schema

OmnXT 规则层已有 `RuleAction::DnsResolver(String)` 和 `RuleAction::Outbound(String)`，但需要定义对外稳定 JSON 结构。

建议对外使用稳定的 snake_case schema，不直接暴露 Rust enum 的默认 serde 外部标签格式：

```json
{
  "id": "openai-us",
  "matcher": {
    "type": "domain_suffix",
    "value": "openai.com"
  },
  "action": {
    "type": "outbound",
    "tag": "unlock:openai:us"
  },
  "enabled": true
}
```

DNS Resolver 建议：

```json
{
  "tag": "dns:cloudflare-doh",
  "proto": "doh",
  "address": "https://cloudflare-dns.com/dns-query",
  "detour": "proxy",
  "timeout_ms": 3000
}
```

Outbound 建议：

```json
{
  "tag": "unlock:netflix:us",
  "type": "node",
  "node_tag": "us-01",
  "health_check_url": "https://www.netflix.com/title/80018499",
  "fail_policy": "fallback_default"
}
```

当前 mobile config 还要补：

- `OmnxtMobileRuleAction` 当前只有 `proxy`、`direct`、`reject`，不能表达 route default action 为指定 outbound 或指定 resolver。
- `OmnxtMobileConfigInput` 虽然包含 `rule_sets`，但对外配置最好不要依赖 Rust enum 默认 serde 形态，避免 Dart/后端生成困难。
- 需要新增 `routing_capabilities` 或 SDK API 版本字段，让 OwlClient 判断当前 native SDK 是否支持 `route_outbound`、`route_dns_resolver`、`route_events`。
- Flutter plugin 需要在 Android/iOS/macOS/Windows 对应 native artifact 更新后再让 OwlClient 使用，否则 Dart 侧生成的新配置会被旧 SDK 拒绝或忽略。
- `crates/omnxt-config` 已有 `EngineConfig.outbounds`、`RoutingConfig.rules`，并且 `validate_engine_config` 会检查重复 outbound tag。SDK 新 schema 应尽量复用或对齐这些底层概念，避免 mobile config、engine config、panel config 三套语义漂移。
- C ABI 更新链路必须列入验收：`crates/omnxt-sdk-cdylib/cbindgen.toml` 只导出 `#[no_mangle]` 函数；新增导出后要重新生成 `omnxt.h` / `libcore.h`，再刷新 `apps/FlutterOmnxtPlugin/android`、`ios/Frameworks/omnxt.xcframework`、`macos/Frameworks/omnxt.xcframework`、`linux`、`windows` 对应二进制和头文件。
- `crates/omnxt-sdk-cdylib/build.rs` 当前调用 cbindgen 失败是 non-fatal warning。发布检查不能只看 `cargo build` 成功，必须 grep 或 diff 生成后的 `omnxt.h` / `libcore.h`，确认新增 routing API、capability、事件函数真的进入头文件。
- Flutter 插件已定义 `android/refresh-android-sdk.sh`、`ios/refresh-ios-sdk.sh`、`macos/refresh-macos-sdk.sh`、`linux/refresh-linux-sdk.sh`、`windows/refresh-windows-sdk.ps1` 等刷新入口；OwlClient 接入 routing 前要确认插件包内 artifact 版本和 `omnxt_mobile_capabilities_json` 返回能力一致。
- iOS NetworkExtension 不是只走 Dart FFI。`apps/FlutterOmnxtPlugin/ios/NetworkExtension/OmnxtPacketTunnelProvider.swift` 通过 `@_silgen_name` 直接绑定 `omnxt_mobile_build_config_json`、`omnxt_mobile_service_auto_select_outbound_json` 等 C ABI。新增或改名 routing C 函数时，Swift bridge、extension target 链接头文件和 xcframework 都要同步检查。
- Android 插件侧 `apps/FlutterOmnxtPlugin/android/src/main/cpp/CMakeLists.txt` 从 `android/src/main/jniLibs/${ANDROID_ABI}` 链接 `libomnxt.so`。刷新 SDK 产物时要确认每个 ABI 目录都有新库，不能只更新本机或临时 `Bin/android/jniLibs`。

### 7.2 连接器注册表

2026-06-26 已完成 SDK 端 connector registry 第二版：移动端 service、local proxy、TUN proxy 已能维护 `outbound tag -> connector` 映射，并在 `RuleAction::Outbound(tag)` 命中时选择指定 connector。第二轮还新增了 `routing_profile.v1` outbounds alias，规则可以引用 `unlock:openai:us` 这类业务 tag，再由 SDK 映射到现有节点 tag。当前仍不引入节点组、External Outbound、health-first 或完整 fail policy 的复杂行为。

完整分流解锁仍需要继续把连接器注册表升级为：

```text
outbound tag -> node config -> connector
```

运行时流程应变成：

```text
请求进入
  -> 构造 MatchContext
  -> PolicyEngine 计算 RuleAction
  -> Direct: 直连
  -> Reject: 拒绝
  -> Proxy: 使用 selected_outbound connector
  -> Outbound(tag): 从 connector registry 选择 tag 对应 connector（第一步已完成）
  -> DnsResolver(tag): 从 dns registry 选择 tag 对应 resolver
```

需要注意：

- `Outbound(tag)` 不是简单改写 `selected_outbound`，当前已按每条连接的 policy decision 选择 connector。
- 未知 outbound tag 当前回退默认 connector，这是保护现有协议的安全策略；正式接入后端 profile 时，还应结合 capability gate 和配置校验减少未知 tag。
- connector 创建后续应支持更细粒度的懒加载和缓存，避免启动时创建大量连接器。
- connector 失败时仍需按后端下发的 fail policy 处理。
- TUN、local proxy、移动端 packet path 已完成第一步一致性接入；TUN DNS 专用路径已接入 resolver registry；fake DNS 回溯和失败回退仍需继续补。

需要重点修改的 SDK 路径：

- `crates/omnxt-sdk/src/mobile.rs`
  - 已新增 route connector registry 构建，并在 local proxy / TUN 启动和热更新时下发 registry。
  - 已接入 `routing_profile.v1` 的 rules、dns_resolvers、outbounds alias 和 capability 声明。
  - 仍需继续接入节点组 outbound、External Outbound、完整 fail policy 和更严格 capability gate。
- `crates/omnxt-local-proxy/src/lib.rs`
  - 已将 `Outbound(tag)` 映射到 registry 中的指定 connector。
  - `DnsResolver(_)` 对 local proxy 的普通连接语义仍保持旧行为；DNS resolver registry 当前主要作用于 TUN DNS detour。
- `crates/omnxt-sdk/src/tun_proxy.rs`
  - 已在 TUN TCP 和非 DNS UDP 代理路径中按 tag 选择 connector。
  - 已补 DNS 路径 resolver 选择和 resolver detour outbound。
  - 仍需补 fake DNS 域名回溯、fail policy 回退和更完整集成测试。
- `crates/omnxt-sdk-cdylib/src/lib.rs`
  - 已新增 `omnxt_mobile_parse_routing_profile_v1_json`。
  - 需要保留旧函数签名兼容已有客户端；新增能力优先走新 JSON 函数或 capability 字段，不要破坏 `omnxt_mobile_parse_config_json` / `omnxt_mobile_build_config_json` 的旧入参。
- `apps/FlutterOmnxtPlugin/lib/src/omnxt_ffi.dart`、`apps/FlutterOmnxtPlugin/lib/src/omnxt_plugin.dart`
  - 当前仓库只有生成 binding 的 export，没有可手改的底层 Dart FFI 源；已同步 iOS/macOS xcframework 头文件声明。
  - 仍需通过正式 binding/artifact 生成流程新增 FFI 绑定、Dart 封装和错误处理。
  - Android/iOS/macOS/Windows/Linux 动态库和头文件必须和 Dart wrapper 同版本提交，否则运行时会出现 symbol not found 或 capability 误判。

节点端需要重点守住边界：

- `crates/omnxt-state/src/lib.rs` 的 `build_engine_load_plan` 已经会把 `working_set.control_plane_outbounds` 合并进 EngineConfig，并按 `client_policy` 生成默认 route。这说明节点端具备旧 control plane outbounds 的基础，但还不是新版 routing profile 编译器。
- `apps/omnxt-node/src/lib.rs` 已有 soga service mode 轮询、ETag/304、`/node`、`/users`、`/alive_ip`、`/traffic`、`/status` 报告链路。若后端要给节点下发新版 routing schema，需要确认 soga/ppanel 两套 service mode 是否都能识别新版字段，并保持 304/hash 兼容。
- P0 最小客户端闭环不建议强行依赖节点端；但标准开发已经确认节点端要消费新版 `routing_profile`、做健康探测并上报，不能把节点端长期排除在开发范围外。
- `crates/omnxt-sdk/src/mobile_dns.rs`
  - 当前有 `MobileDnsUpstreamMode::Direct/Proxy/Policy`，但还没有“指定 resolver tag”的完整 registry。
  - 需要支持 `DnsResolver(tag)` 对应不同 upstream/resolver/detour。

### 7.3 DNS 调度

DNS 分流需要单独实现：

- 域名解析前能匹配 DNS 规则。
- 解析器可以选择 direct/proxy/outbound detour。
- fake DNS 场景下需要保留域名到 IP 的映射，后续连接仍可回到原始域名规则。
- DNS 结果应有缓存和 TTL。
- DNS 查询失败要有 fallback。

对 OpenAI、Netflix 这类服务，DNS 路径和连接路径可能都影响结果，所以 DNS Resolver 不能只是配置项，需要进入实际解析流程。

### 7.4 RuleAction 运行时闭环

需要逐条确认以下链路：

- `RuleAction::Direct`：已有直连行为，需验证 TUN 和 local proxy 一致。
- `RuleAction::Proxy`：使用当前默认 selected outbound。
- `RuleAction::Reject`：连接拒绝并返回明确错误。
- `RuleAction::DnsResolver(tag)`：DNS 查询使用指定 resolver，不应只是当成 proxy。
- `RuleAction::Outbound(tag)`：流量使用指定 outbound connector，不应只是当成 proxy；SDK local proxy / TUN 第一版已完成该闭环。

当前剩余重点不是再新增 `Outbound` 枚举，而是补齐以下闭环：

- `DnsResolver(tag)` 进入 resolver registry，并支持 DoH / DoT / UDP / TCP、detour 和 fallback。
- `Outbound(tag)` 在 connector 失败时按 `fail_policy` 回退，而不是只依赖未知 tag 回退默认 connector。
- route decision、fallback、health changed 等事件要能被 C ABI / Flutter plugin / OwlClient 诊断页消费。
- `routing_profile.v1` 要经过 schema 校验和 capability gate 后再进入 SDK 执行，避免旧 SDK 或不支持的 External Outbound 被静默执行。

### 7.5 状态和事件

OmnXT 需要补可观测事件：

- 命中的规则 ID。
- 匹配条件。
- 选择的 action。
- 选择的 outbound tag。
- 选择的 DNS resolver tag。
- fallback 是否发生。
- 失败原因。
- 连接延迟。
- 健康状态变化。

建议新增事件：

- `route_decision`
- `route_fallback`
- `outbound_health_changed`
- `dns_resolver_health_changed`
- `route_config_applied`
- `route_config_rejected`

OwlClient 的诊断页可以直接消费这些事件。

### 7.6 配置热更新

需要支持：

- 后端策略 hash 改变后，客户端刷新配置。
- OmnXT 应用新规则时不中断已建立连接，或者提供明确的重启策略。
- 出站健康状态更新时，不需要重启整个服务。
- 配置失败时保留上一份可用配置。

### 7.7 测试要求

OmnXT 必须补测试：

- `RuleAction::Outbound(tag)` 能选择正确 connector。第一步已补 mobile registry 命名节点命中测试，后续还要补 local proxy / TUN 集成测试。
- `RuleAction::DnsResolver(tag)` 能选择正确 resolver。
- TUN TCP、TUN UDP、local SOCKS、local HTTP 行为一致。
- fake DNS 场景下域名规则不会丢失。
- 出站失败能按 `fail_policy` 回退。
- 配置热更新失败不会清空旧配置。

### 7.8 节点端是否需要修改

节点端是否修改取决于产品范围。

第一阶段客户端分流解锁：

- 不要求远端 OmnXT 节点协议变化。
- 不要求节点端理解每个用户的 OpenAI/Netflix 域名规则。
- 节点只需要继续按现有协议承载连接。
- 需要后端给客户端下发节点能力和 routing profile，客户端 SDK 自己执行分流。

节点控制面增强：

- 标准开发目标已确认使用新版 `routing_profile`，不能只保留旧 `dns/outbound/block` 兼容。旧结构继续保留用于旧节点和回滚，新节点必须能消费新版 schema。
- 节点端需要修改 `crates/omnxt-control/src/ppanel/models.rs`、`assembler.rs` 和 `crates/omnxt-state/src/lib.rs` 的 bundle 消费逻辑，支持 routing profile / rule / resolver / outbound / unlock service / health 字段。
- 节点端需要做 Outbound 健康检查、解锁服务探测、自动禁用或降权不可用出站，并新增状态上报字段和后端接收接口。
- 如果要服务端侧二次分流，例如节点收到用户流量后再按目标域名选择上游 outbound，则节点运行时也要接入 policy route planner 和 outbound registry。这是更大的功能，不应混入第一阶段客户端闭环。

建议当前优先级：

```text
P0/P1: 后端 schema + OwlClient 拉取 + SDK capability gate
P2/P3: OmnXT SDK/local proxy/TUN 真正执行 Outbound/DnsResolver
P4: 后端/客户端健康检查与 fallback
P5: 服务预设和规则更新
P6: 节点端 control/ppanel 新 schema、节点健康探测和上报
```

一句话结论：必须先改 SDK 端保证客户端可执行；标准开发不能漏节点端，新版 `routing_profile`、节点端健康探测和状态上报都要进入正式开发范围。

## 8. OwlClient 需要修改的部分

### 8.1 API 数据模型

客户端需要新增模型：

- `RoutingProfile`
- `RoutingRule`
- `RoutingMatcher`
- `RoutingAction`
- `DnsResolver`
- `RouteOutbound`
- `UnlockService`
- `RouteConfigVersion`

结合现有 OwlClient 代码，还需要新增接入层：

- `core/api/api_client.dart`：新增获取 routing config 的 HTTP 方法；这是所有面板共用抽象，不能只在 `PpanelApiClient` 单独加方法。
- `core/api/api_paths.dart`：如果是 NPanel/PPanel 官方接口，补充 `/v1/public/routing/config` 等路径常量。
- `core/api/panels/ppanel/ppanel_api_client.dart`：实现官方 NPanel/PPanel routing config 拉取。
- `core/api/panels/xboard/*`、`core/api/panels/xiaov2board/*`：至少提供 unsupported/fallback 行为，避免抽象方法新增后其它面板实现编译失败。
- `PanelType` 实际包含 `ppanel`、`xboard`、`xiaov2board`、`v2board`、`sspanel`。其中 `v2board` 当前复用 `Xiaov2boardApiClient`，`sspanel` 多处临时复用 PPanel 逻辑。routing 方法新增时不能只覆盖 xboard/xiaov2board，还要明确 `v2board` 和 `sspanel` 是 unsupported、fallback，还是复用哪一套实现。
- `lib/core/config/panel_type.dart` 已有 `FeatureFlags.fromPanelType`。routing 能力建议进入 feature flag 或专门 capability，而不是在 UI、repository、connection service 里散落 `PanelConfig.current == PanelType.ppanel` 判断。
- `features/subscription/data/subscription_repository.dart` 或新增 `features/routing/data/routing_repository.dart`：封装 routing config 拉取。
- Riverpod provider：管理 loading、error、config、hash、lastUpdated。
- 本地缓存：保存最后一份可用 routing config，后端不可用时可以降级使用。当前项目已有 `StorageKeys` 和 Drift `AppDatabase`，如果 routing config 放 SecureStorage，需要新增 key；如果放 SQLite，需要新增表、DAO、`schemaVersion`、migration 和 `app_database.g.dart`。
- 能力协商：请求中带客户端版本、OmnXT SDK 版本、平台、支持的 routing capability。
- 失败降级：routing config 拉取失败时，不应影响原有节点连接；应退回当前本地 `proxyMode` + CN/private direct 规则。
- `core/api/api_provider.dart` 按 `PanelType` 返回 `PpanelApiClient`、`XboardApiClient`、`Xiaov2boardApiClient`，新增 routing 方法后必须确认每个分支都有明确行为。即使 routing 只支持 NPanel/PPanel，也要让其它面板返回 unsupported/fallback，而不是编译期漏实现。
- 现有测试里有多个 `_FakeApiClient implements ApiClient`。`ApiClient` 新增抽象方法后，`test/account_scoped_state_test.dart` 这类完整 fake 需要补方法；使用 `noSuchMethod` 的 fake 可以继续兜底，但要新增 routing repository/provider 测试覆盖成功、unsupported、后端失败和缓存回退。
- 如果新增客户端页面或诊断入口，还要同步 `lib/core/routes/routes.dart`、`lib/core/routes/app_router.dart`，并确认深链、返回键、登录态重定向不会绕过 routing capability gate。

现有节点模型已有国家、城市、标签、协议等信息，可以继续复用，但需要增加：

- 节点支持哪些解锁服务。
- 节点是否可作为专用 Outbound。
- 当前用户可用哪些 Profile。
- 当前客户端是否支持高级 routing。

`Node` 模型需要谨慎扩展：

- `tags` 现在已用于节点分组/筛选，不宜混入所有解锁能力。
- 可以新增 `capabilityTags`、`unlockServices`、`routeRole`、`routeWeight`，或在 routing config 中单独维护 node tag 到 capability 的映射。
- `protocols` 当前是原始订阅链接/协议 JSON 字符串，不能把 routing JSON 混进去，否则会污染连接配置解析。

### 8.2 OmnxtConfigBuilder 改造

`OmnxtConfigBuilder` 需要从“本地内置少量规则”升级为“后端 Profile 驱动”：

- 输入后端下发的 `routing_profile`。
- 生成 OmnXT `rule_sets`。
- 生成 OmnXT `dns` 配置。
- 生成 OmnXT `outbounds` 或等价节点引用。
- 生成 `geoip_entries`、`geosite_entries`。
- 保留当前 CN direct 逻辑作为默认 Profile 或 fallback。

当前 `_buildRouteRules` 的 `default_action` 逻辑还比较简单，需要支持：

- `default_action`
- `final_outbound`
- `fallback_action`
- `observe` 模式。
- 多 Profile 合并后的最终规则。

还需要注意现有启动链路：

- `OmnxtConnectionService.connect` 现在先 `prepareRules()`，再按 `proxyMode`、`countryCode` 注入 `ruleSets`。
- routing config 必须在 `connect` 构建 config 之前完成拉取、缓存读取和 schema 校验。
- `switchOutbound` 当前调用 `selectOutbound(selectedTag)`，它是手动切换默认出站；分流解锁不能依赖每条连接前调用 `selectOutbound`，必须由 OmnXT 运行时按规则选择。
- `usesTunForProxyMode` 决定不同平台是否启用 TUN。routing config 需要明确 desktop local proxy 和 mobile TUN 的差异，避免 Android/iOS 生效而 macOS/Windows 不一致。
- `assets/omnxt_rules/` 已在 `pubspec.yaml` 中声明。新增 geosite/geoip 或远程规则缓存目录时，要同步 asset、下载目录、hash 文件和打包脚本，避免 debug 有文件、release 缺资源。

### 8.2.1 OwlClient 生成物和构建链

OwlClient 有多类生成物，routing 模型落地时不能只改手写 Dart：

- Freezed/JSON 模型：新增 `part '*.freezed.dart'`、`part '*.g.dart'` 后运行 `dart run build_runner build --delete-conflicting-outputs`。
- Drift：新增表或字段后同步 `schemaVersion`、migration、DAO，并更新 `lib/core/storage/app_database.g.dart` 和相关 `*_dao.g.dart`。
- Slang/i18n：项目使用 `slang_build_runner`，如果客户端 UI 增加 routing 文案，需要运行 `dart run build_runner build --delete-conflicting-outputs` 生成本地化代码，避免 release 缺 key。
- FlutterOmnxtPlugin：OmnXT SDK 改动后，需要在 `third_party/OmnXT/apps/FlutterOmnxtPlugin` 内刷新 Android `jniLibs`、iOS/macOS `xcframework/dylib`、Windows `dll`、Linux `so` 等产物，再让 OwlClient 更新 `third_party/OmnXT` gitlink。
- Android 打包脚本当前优先使用 `third_party/OmnXT/apps/FlutterOmnxtPlugin/android/src/main/jniLibs`，所以发布前要确认提交的是插件目录里的二进制，而不是仅本地 `Bin/android/jniLibs`。
- 现有测试名要纳入开发清单：`test/omnxt_config_builder_test.dart` 覆盖 config builder 输出，`test/omnxt_connection_service_test.dart` 覆盖连接启动链路，`test/domain_manager_test.dart` 覆盖不同 PanelType 的接口路径和响应形态。routing 接入后这三类测试都应补成功、unsupported、缓存回退或路径差异用例。

### 8.3 OmnxtRuleService 改造

当前规则资源主要是 CN，需要扩展：

- 支持服务级 geosite 包。
- 支持远程规则更新。
- 支持后端下发自定义域名规则。
- 支持规则版本和 hash。
- 支持规则资源缺失时降级。

建议资源来源优先级：

```text
后端下发自定义规则
  > 客户端缓存规则包
  > App 内置基础规则
  > 最小 fallback 规则
```

规则资源还要处理：

- `assets/omnxt_rules` 目前由 asset manifest 复制到可写目录，后续远程规则包需要有下载目录、版本文件和校验 hash。
- geosite/geoip 文件和后端下发的 inline rules 要区分来源，便于诊断。
- 如果后端下发了 `geosite:openai` 但本地没有对应资源，应在诊断中明确提示“规则资源缺失”，而不是静默走默认代理。

### 8.4 客户端诊断和统计接入

现有 `OmnxtPluginClient` 已有 `testRule`、`ruleSetCount`、`groups`、`activeGroup`、`selectOutbound` 等接口。规划需要新增或复用诊断入口：

- 启动前：展示 routing config hash、Profile code、规则数量、资源包版本。
- 启动后：调用 `testRule(domain)` 展示命中规则。
- 运行中：从 OmnXT 事件或 stats 中读取 `route_decision`、`selected_outbound`、`default_action`。
- 切换节点后：确认默认 `selected_outbound` 改变，但 per-rule outbound 不应被覆盖。
- 日志上报：把 routing config 解析失败、规则资源缺失、SDK 不支持 capability 等错误打到可诊断日志。

建议新增或扩展：

- 设置页：分流模式
  - 关闭
  - 智能分流

### 8.6 客户端安全策略

- 默认不启用会改变用户网络行为的大范围规则。
- 默认保留 CN direct / private direct。
- 解锁规则先支持 observe 模式，确认命中后再 enforce。
- 不在客户端硬编码服务可用性结论。
- 不根据节点名称显示“支持 Netflix”，必须来自后端 capability 或测试结果。
- SDK capability 不满足时必须 fail closed 到“只展示/不执行”，不能静默把高级规则当普通 proxy 处理。
- routing config hash、profile code、SDK capability、规则资源版本要写入诊断日志，便于客户现场复现。

## 10. 下发配置示例

下发配置的唯一冻结标准见 `4.1 routing_profile v1 冻结契约`，完整 JSON 示例见 `4.1.13 完整 v1 示例`。

后续后端 proto、OpenAPI、OmnXT SDK schema、Flutter plugin FFI、OwlClient model/cache/diagnostic、节点端 control/state 适配都以 `schema = routing_profile.v1` 为准。不要再基于旧示例单独实现 `routing_version/mode/profile/dns_resolvers/outbounds/rules` 的简化结构，避免出现后端、SDK、客户端三套字段语义漂移。

### P4：健康检查和自动降级

目标：解锁节点不可用时自动降级。

工作：

- 后端健康检查：负责全局 resolver/outbound/service 探测、连续失败计数、自动禁用或降权建议。
- 客户端健康检查：负责当前设备到候选 outbound 的实时 RTT、URL test、DNS 可达性和最近失败原因。
- 节点端健康检查：负责节点本机对 External Outbound、DNS Resolver、解锁服务 URL 的探测和状态上报。
- OmnXT 运行时 fallback。
- Outbound 自动禁用或降权。
- 默认 fail policy 按保守降级执行：解锁服务默认 `fallback_default`，敏感服务可 `fail_closed`，直连只用于明确 direct 规则。

验收：

- 某个解锁出站不可用时，不影响其他普通代理流量。
- 管理后台能看到失败原因。
- OwlClient 能看到本机 RTT/可达性和后端全局健康状态的差异。
- 节点端能上报 resolver/outbound/service health，后端能聚合展示。

## 13. 风险点

### 13.1 OmnXT 规则语言与运行时不等价

此前 `RuleAction::Outbound` 已存在但运行时不一定能按规则切换连接器。2026-06-26 SDK 第一版已经补上 connector registry 和 local proxy / TUN 的 `Outbound(tag)` 选择闭环，最大风险已从“完全不执行”下降为“执行范围还不完整”。

剩余风险：

- `DnsResolver(tag)` 仍未进入 resolver registry。
- `Outbound(tag)` 失败回退还没有按后端 `fail_policy` 执行。
- `routing_profile.v1` schema、capability gate、C ABI / Flutter plugin 还没有完整发布。
- External Outbound、节点组 outbound、health-first / lowest-latency 等选择策略还未实现。

### 13.2 DNS 与连接路径割裂

如果 DNS 走直连，连接走代理，或者 DNS 走错误地区，解锁可能失败。DNS Resolver 必须和路由策略一起设计。

### 13.3 客户端误展示

不能因为节点名包含 `Netflix` 就显示支持 Netflix。必须来自后端配置或健康测试。

### 13.4 传统订阅污染

高级 routing 字段不能直接下发给不支持的客户端，否则可能导致订阅解析失败。

### 13.5 规则维护成本

流媒体和 AI 服务域名变化快，后续需要服务预设和规则更新机制，不能全靠手动填域名。

### 13.6 生成链遗漏

NPanel 后端和前端都有生成代码。只改 proto/ent/source，不更新 pb、wire、swagger、frontend service，会导致本地能编译的模块和前端实际调用不一致。

### 13.7 订阅应用 UA 命中错误

当前订阅应用是子串匹配，并且会选中第一个命中的应用。如果新增 OwlClient/OmnXT 应用行顺序或 UA 过于宽泛，可能导致官方客户端拿到普通订阅，或者普通客户端拿到增强模板。

### 13.8 节点 tags 语义过载

当前节点 tags 已用于套餐、节点筛选、节点组辅助逻辑。若直接把 `netflix`、`openai` 等解锁能力塞进 tags，可能影响套餐可见节点范围。解锁能力应优先独立字段化。

## 15. 已确认产品决策

以下 10 项已经确认，可作为后续标准开发输入：

1. 分流策略作用范围做到用户级。最终优先级为：用户专属 Profile > 套餐 Profile > 用户组 Profile > 节点 Profile > 全局默认 Profile。
2. 解锁服务通过配置扩展，不写死在代码里。第一版可以内置一批默认预设，但新增、修改、禁用服务都应通过数据库或配置完成。
3. Outbound 同时支持两种形态：默认复用现有节点 / 节点组作为 outbound，同时允许配置 External Outbound。External Outbound 第一批支持 `WireGuard`、`SOCKS`、`HTTP`。
4. DNS Resolver 第一版支持 `DoH` + `DoT`，UDP/TCP 作为兼容和 fallback。
5. OwlClient 服务解锁策略由后台控制。客户端负责展示状态、诊断、测试和执行，不提供用户本地覆盖后台策略的开关。
6. 失败回退采用保守降级：解锁服务默认 `fallback_default`，即主 outbound 不可用时回到同服务候选池或当前默认代理；敏感服务可配置 `fail_closed`；`fallback_direct` 只用于明确 direct 场景，不作为解锁服务默认值。
7. OmnXT 对外 JSON schema 采用稳定 snake_case action，不直接暴露 Rust enum serde 格式。
8. 健康检查两端都做。后端负责全局健康、连续失败、禁用/降权建议；客户端负责当前设备实时 RTT、URL test、DNS 可达性，避免配置下发后无法感知现场网络状态。
9. 节点端第一版支持新版 `routing_profile`，旧 `dns/outbound/block` 只作为兼容和回滚路径。
10. 节点端需要增加解锁服务探测和健康状态上报，后端要能接收、聚合并展示节点 resolver/outbound/service health。

## 16. 标准开发规划

后续开发按“先契约、再后端、再客户端消费、再 OmnXT 执行、最后健康闭环和节点增强”的顺序推进。不要先堆 UI，也不要先把服务域名硬编码到客户端。

### 16.1 第一阶段：契约冻结和最小闭环

目标是在不影响真实用户流量的情况下冻结 schema 和能力边界。

必须完成：

- 定义 `routing_profile`、`routing_rule`、`dns_resolver`、`route_outbound`、`unlock_service` 的 JSON schema。
- schema 必须覆盖用户级作用域、DoH/DoT、External Outbound、snake_case action、fallback policy、health status。
- 后端提供固定或数据库驱动的 preview config。
- OwlClient 能拉取、缓存、展示 routing hash 和 Profile，但默认不执行。
- OmnXT SDK 能解析 schema，并报告 capability。
- 节点端确认新版 `routing_profile` 下发入口和 health 上报契约。

验收标准：

- 输入 `openai.com` 能在后端 preview 中看到命中规则、action、resolver、outbound、fallback。

- 后端全局 health worker。
- 节点端新版 `routing_profile` 下发。
- 节点 health 上报接收接口。

验收标准：

- Admin 可以配置完整 Profile，并预览某用户最终配置。
- OwlClient 可拉取增强配置。
- 新 OmnXT 节点可拉取新版 routing profile。
- 旧客户端、旧节点仍兼容。

### 16.3 第三阶段：OwlClient 消费和诊断

- 后端不可用时客户端仍能按现有节点正常连接。
- 客户端能明确展示后台 health、节点 health、本机 health。
- 用户不能开启后台未授权的服务解锁。

### 16.4 第四阶段：OmnXT SDK 真正执行

目标是让规则命中后真的按 tag 选择 connector 和 resolver。

必须完成：

验收标准：

- `openai.com` 命中 `unlock:openai:us` 并实际走指定 connector。

### 16.5 第五阶段：健康闭环和自动降级

必须完成：

- 预设灰度、回滚和审计。

验收标准：

- 管理员可以不改代码新增服务预设。
- 预设更新会触发 routing hash 和客户端刷新。
- 服务级测试报告能指导 outbound 调整。

## 17. P0 / P1 / P2 落地状态和下一步推荐

### 17.1 P0 已完成

### 17.3 P2 已完成

本轮 P2 把“Admin 配置”安全接到“公共只读配置 + OwlClient 消费”，但仍不执行真实流量切换：

- 后端公共只读接口改为复用 P1 的同一套 routing 编译器：
  - `GET /v1/public/routing/config`
  - `POST /v1/public/routing/preview`
- 公共接口优先读取数据库启用 Profile / Rule / DNS Resolver / Outbound / Unlock Service。

- 公共 routing config 增加短 TTL 缓存，并继续支持 `If-None-Match` / `routing_hash`。
- 新增后端单测覆盖：
  - 空库回退 P0 fixture。
  - 规则引用缺失 outbound 时回退 P0 fixture 并返回编译错误。

### 17.4 推荐下一步

下一步建议进入 P3：做 ppanel-only 的 OmnXT overlay / capability 联调，但仍用显式开关保护真实执行。

1. OwlClient 先做执行前 capability gate：
   - 只在 `PanelType.ppanel && supportsRoutingProfile` 下启用。
   - 校验后端 `required_features`、OmnXT SDK feature、Flutter plugin feature 三方能力。
   - 不满足能力时继续只展示，不执行。

2. OmnXT 配置 overlay 先做 dry-run：
   - 把 routing profile 编译成 OmnXT overlay 输入。
   - 输出本地 dry-run 决策日志：domain、rule、resolver、outbound、fallback。
   - 与后端 Public preview 对比 `openai.com` 命中结果。

3. Admin 再补最小执行开关：
   - Profile `mode=observe/enforce` 保持后端可控。
   - 默认继续 `observe`。
   - 只有 ppanel 客户端、能力满足、Profile enforce 三者同时满足才允许进入真实执行。

P3 验收硬线：其它面板仍不请求 routing 接口；observe 模式绝不改变真实连接；enforce 需要可回滚、可诊断、可解释。
